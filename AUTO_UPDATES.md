# Auto-Updates Setup Guide

This guide explains how auto-updates work in Meeting Notes using `electron-updater`.

## How It Works

Meeting Notes uses `electron-updater` to automatically check for and install updates from GitHub Releases.

### Update Flow

1. **Check for Updates**: App checks GitHub Releases every hour (and 3 seconds after startup)
2. **Update Available**: User sees a dialog asking if they want to download
3. **Download**: Update downloads in the background
4. **Install**: User is prompted to restart the app to install the update

### User Experience

- Non-intrusive: Users can choose to update now or later
- Background download: Updates download while the app is running
- Auto-install on quit: Updates install automatically when the app closes
- No update? No interruption: Users won't see any dialogs if they're on the latest version

## Configuration

### package.json

The `publish` configuration tells electron-updater where to check for updates:

```json
{
  "build": {
    "publish": [
      {
        "provider": "github",
        "owner": "deanhume",
        "repo": "meeting-notes"
      }
    ]
  }
}
```

### Update Checking Frequency

By default, the app checks for updates:
- 3 seconds after startup
- Every 60 minutes while running

To change the frequency, edit `main.js`:

```javascript
// Check every 2 hours instead
setInterval(() => {
  autoUpdater.checkForUpdates();
}, 2 * 60 * 60 * 1000);
```

## Publishing Updates

### Step 1: Build Your App

```bash
npm run build        # Windows
npm run build:mac    # macOS
npm run build:all    # Both platforms
```

This creates installer files in the `dist/` folder.

### Step 2: Create a GitHub Release

1. Go to https://github.com/deanhume/meeting-notes/releases
2. Click "Draft a new release"
3. Create a new tag (e.g., `v1.1.8`)
4. Title it (e.g., "Version 1.1.8")
5. Add release notes describing what changed
6. Upload the installer files:
   - `Meeting Notes Setup 1.1.8.exe` (Windows)
   - `Meeting Notes-1.1.8.dmg` (macOS)
   - `Meeting Notes-1.1.8-mac.zip` (macOS alternative)
   - `latest.yml` (Windows - auto-generated by electron-builder)
   - `latest-mac.yml` (macOS - auto-generated by electron-builder)

7. Click "Publish release"

**Important:** You MUST upload the `.yml` files for auto-updates to work! These files are automatically created by electron-builder in the `dist/` folder.

### Step 3: Users Get the Update

Once published, users running the app will:
1. Get a notification within an hour (or next time they start the app)
2. Choose to download the update
3. Restart to install

## File Types for GitHub Releases

### Windows
- `Meeting Notes Setup x.x.x.exe` - The installer users download
- `latest.yml` - **Required** for auto-updates (tells the updater about the new version)

### macOS
- `Meeting Notes-x.x.x.dmg` - DMG installer for users
- `Meeting Notes-x.x.x-mac.zip` - ZIP alternative
- `latest-mac.yml` - **Required** for auto-updates

## Testing Auto-Updates

### In Development
Auto-updates are disabled when running in development mode (`npm start`). You'll see:
```
Running in development mode - auto-updates disabled
```

### Testing with a Real Build

1. Build your current version:
   ```bash
   npm run build:dir
   ```

2. Run the built app from `dist/win-unpacked/Meeting Notes.exe`

3. Create a fake GitHub release with a higher version number

4. The app should detect the update

**Better approach:** Use electron-updater's dev mode:

```javascript
// In main.js, add this for testing:
if (!app.isPackaged) {
  autoUpdater.updateConfigPath = path.join(__dirname, 'dev-app-update.yml');
}
```

Create `dev-app-update.yml`:
```yaml
provider: github
owner: deanhume
repo: meeting-notes
```

## Automatic vs Manual Updates

### Current Setup (Automatic - Recommended)
- ✅ Users are notified when updates are available
- ✅ Users choose when to download
- ✅ Updates install on quit automatically
- ✅ No action needed from users after initial download

### Alternative: Silent Updates
If you want completely silent updates, change in `main.js`:

```javascript
autoUpdater.autoDownload = true;  // Auto-download without asking
autoUpdater.autoInstallOnAppQuit = true;  // Auto-install when app closes

autoUpdater.on('update-downloaded', () => {
  // Silently install on next restart - no dialog
});
```

### Alternative: Manual Check Only
Add a "Check for Updates" menu item:

```javascript
const { Menu } = require('electron');

const menu = Menu.buildFromTemplate([
  {
    label: 'Help',
    submenu: [
      {
        label: 'Check for Updates',
        click: () => {
          autoUpdater.checkForUpdates();
        }
      }
    ]
  }
]);

Menu.setApplicationMenu(menu);
```

## Troubleshooting

### "No updates available" but I know there is one
- Check that `latest.yml` (Windows) or `latest-mac.yml` (macOS) is uploaded to the release
- Verify the version number in package.json is higher in the new release
- Check the GitHub repo/owner matches your config

### Updates not downloading
- Check your internet connection
- Look in the logs: `%APPDATA%/meeting-notes/logs/main.log` (Windows) or `~/Library/Logs/meeting-notes/main.log` (macOS)
- Ensure the installer files are attached to a published (not draft) release

### Error: "Cannot find latest.yml"
- The YML files are auto-generated by electron-builder in the `dist/` folder
- You must upload them to GitHub Releases along with the installers

### Users on old version not getting updates
- They need to manually download the new version once to get the auto-updater
- Auto-updates only work for versions that have electron-updater installed

## Version Management

When you run `npm run build`, the version automatically increments by 0.0.1.

To manually set a version:
```bash
npm version 1.2.0 --no-git-tag-version
npm run build
```

## Best Practices

1. **Always test builds before releasing** - Run the built app locally
2. **Write clear release notes** - Users appreciate knowing what changed
3. **Upload all files** - Don't forget the `.yml` files!
4. **Tag your releases** - Use semantic versioning (v1.2.3)
5. **Keep old releases** - Don't delete them; users might need to downgrade

## Security

- electron-updater uses HTTPS to download updates
- Code signing ensures updates haven't been tampered with (see BUILD_MAC.md for macOS signing)
- Users can verify downloads by checking GitHub Releases manually

## Logs

Auto-update logs are stored at:
- **Windows**: `%APPDATA%\meeting-notes\logs\main.log`
- **macOS**: `~/Library/Logs/meeting-notes/main.log`

Check these logs if you need to debug update issues.

## Disabling Auto-Updates

If you want to disable auto-updates entirely, comment out in `main.js`:

```javascript
// setupAutoUpdater();
```
